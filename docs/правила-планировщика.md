# Правила работы планировщика (MVP)

Документ для согласования с директором. Описывает **фактическую логику кода** (жадный планировщик `GreedyScheduler`, метрики `ScheduleMetrics`, закрытие смен).

**Сопровождение:** при изменении логики планировщика, API или формата `data/schedule.json` этот файл обновляется **в том же PR/коммите**, что и код. См. также [README.md](../README.md) и [согласование-с-директором.md](согласование-с-директором.md) (ручная проверка на встрече).

*Последняя сверка с кодом: май 2026.*

---

## 1. Назначение

- Строит **плановое** расписание: какой заказ, какая деталь (штука), какая операция, на каком станке, с какого по какое время.
- Сохраняет результат в `data/schedule.json`.
- **Не** пересчитывает уже запланированные заказы при **добавлении нового** (`POST /orders`): новый заказ только дописывается в план с учётом занятости станков и ограничений очереди.
- **Пересчитывает** будущий хвост (`PLANNED`) у затронутых заказов при **закрытии смены** (`POST /shifts/close`) — см. §9.

---

## 2. Входные данные

| Источник | Что задаёт |
|----------|------------|
| `partDefinitions` | Тип детали, **приоритет** (число), цепочка операций: `taskId`, `sequence`, длительность (`PT…`), требуемая возможность станка (`TURNING`, `MILLING`, …) |
| `machines` | Станки, группа, возможности, статус, `availableAt` |
| `machineGroups` | Смены по дням недели (`workWindows`), время **переналадки** (`setupMinutes` и/или `setupDuration`) |
| `lastClosedShiftEndByGroup` | До какого момента смены группы уже «закрыты» мастером |
| `simulationClock` | Включена ли симуляция времени и текущее «сейчас» |
| `scheduling` | Опции планировщика: `overlapBatches` (по умолчанию `false`) — перекрытие пакетов между станками |
| Заказ `POST /orders` | Опционально `orderId`; список `{ partId, quantity? }`; технология из справочника |

**Валидация перед планированием (`OrderValidator`):**

- В заказе хотя бы одна позиция, без дубликатов `partId`.
- `partId` обязателен; `quantity` если указан — ≥ 1 (иначе по умолчанию 1).
- Каждый `partId` есть в справочнике.
- Для **каждой** операции детали есть хотя бы один **работоспособный** станок (`isOperational`) с нужной `capability`.
- `orderId` ещё не существует в системе (если передан явно).

**Автонумерация заказа:** если `orderId` в запросе пустой, присваивается **`З-ГГГГ-NNNN`** (`OrderIds`): год — по часовому поясу завода (Москва), номер — следующий за существующими с тем же годом в формате `З-YYYY-NNNN`.

## 3. Время и очередь заказов

1. **`createdAt` заказа** = текущее время планировщика в момент `POST /orders` (симуляция или системные часы).
2. **`priority` заказа** — чем **больше** число, тем раньше заказ в очереди планирования. По умолчанию из `createdAt` (более ранний заказ → выше приоритет); в JSON можно сохранить другое положительное значение (`OrderPriorities.resolve`).
3. **Старт планирования заказа** (`orderStart`):
   - не раньше `factoryStartedAt`;
   - не раньше `createdAt` заказа;
   - не раньше **текущего «сейчас»** (если симуляция уже ушла вперёд).
4. **Очередь между заказами — по `priority`:** заказ с меньшим приоритетом **не имеет права** увеличить `readyAt` заказа с большим приоритетом. Проверка — симуляцией каждого шага назначения в `GreedyScheduler`.
5. **Симуляция времени** (`PUT /time`): только **вперёд**; откат запрещён (`SchedulingException`).
6. При `simulationClock.enabled: false` сдвиг времени через API запрещён; «сейчас» = системные часы (`SystemCurrentTimeProvider`).

**Побочный эффект времени:** при `GET /shifts/context` и `PUT /time` вызывается **автозакрытие пустых смен** (§9.1).

## 4. Технология: пакет по операциям (главное правило цеха)

Для **каждого типа детали** в заказе (своё `quantity`, своя цепочка операций):

### 4.1. Между операциями (по `sequence`)

- Сначала **все штуки** проходят операцию с `sequence = 0`.
- Затем **все штуки** — операцию с `sequence = 1`, и т.д.
- Операция `N+1` для **любой** штуки не начинается, пока **все** штуки этого типа не **запланированы или выполнены** по операции `N`, кроме случаев overlap (§5).

### 4.2. Внутри одной операции (один `taskId`)

- Штуки обрабатываются **строго по порядку** `unitIndex`: 0 → 1 → 2 → … → `quantity − 1`.
- Штука `k` не получает эту операцию, пока не запланирована/выполнена та же операция для штуки `k − 1`.

### 4.3. Что это **не** означает

- Это **не** поштучный маршрут «штука 1 прошла все станки, потом штука 2».
- Это **не** смешивание операций между штуками на **одном** станке. При **≥2 станках** одной capability чистовая на станке B может начаться, пока черновая ещё идёт на станке A (§5).

## 5. Несколько типов деталей в одном заказе

| Правило | Поведение |
|---------|-----------|
| Приоритет детали | Из `partDefinitions.priority` (**только внутри заказа**) |
| Основной шаг | Сначала деталь с **большим** приоритетом, у которой есть «готовая» работа (`readyWork`) |
| Параллельность | Детали с **меньшим** приоритетом могут планироваться на других станках, если не сдвигается `partReadyAt` более приоритетных деталей **в том же заказе** |
| Между заказами | Новый заказ не ухудшает `readyAt` более ранних по `priority` |

**Склейка** одинаковых деталей из разных заказов на одном станке без переналадки **не выполняется**.

### Перекрытие пакетов (опционально)

**По умолчанию выключено** (`scheduling.overlapBatches: false` в JSON). Включение: `"scheduling": { "overlapBatches": true }` или `-Dscheduler.overlapBatches=true` (свойство JVM **перекрывает** JSON).

При **выкл.** — только классический пакет: все штуки op.N, затем все op.N+1; на одном станке — полный пакет op.N.

При **вкл.** — согласованная логика overlap (ниже). Overlap только если op.N и op.N+1 **можно развести на разные станки** (≥2 машин в пуле — планируем на разных; 0 общих — разные capability). **Один** общий станок — классический пакет.

**Для каждой штуки k:** `start(op N+1, k) ≥ end(op N, k)` (учитывается в `previousTaskEnd`).

**Пакет op.N+1** на **одном** станке: штуки **0→1→…→Q−1** **без пауз**; переналадка только перед первой.

| Режим | Когда | Старт пакета op.N+1 (штука 0) |
|-------|--------|-------------------------------|
| `PIPELINE` | ≥2 станка одной capability; фреза→расточка и т.п. | Если весь op.N в плане: `max_k(end(N,k) − k·p_{N+1})`; иначе `end(N,0)` |
| `PER_UNIT` | Чистовая токарка → шлифование (разные цеха) | `end(N,0)` для шт.0; шт.k: `max(end(N,k), end(N+1,k−1))` |
| `NONE` | Одна машина на обе op | После **всего** пакета op.N |

- **Две токарки:** чистовая #k на 03 не раньше конца черновой #k на 02; пакет чистовой на 03 может стартовать до конца всей черновой на 02, если непрерывность и подача по штукам соблюдены.
- **Чистовая → шлифование:** без пакетного конвейера; шлифование #k только после конца чистовой #k.
- На **том же** физическом станке для op.N и op.N+1 — сначала весь пакет op.N на этом станке.

## 6. Как выбирается следующая работа

Планировщик добавляет заказ **по одному назначению** (рабочая операция [+ переналадка]) за итерацию.

1. **Основной кандидат** — первая по приоритету деталь с `readyWork`.
2. Проверки внутри заказа и очереди заказов → назначение.
3. Иначе **параллельный** режим для низкоприоритетных деталей без ухудшения `partReadyAt` важнее стоящих.
4. Нет прогресса → `Cannot schedule order`.

После планирования — у каждой детали все операции на все `quantity` штук (`verifyOrderFullyScheduled`).

## 7. Выбор станка и момент начала

1. Станки: не `DOWN` / `MAINTENANCE` / `SETUP`, есть нужная `capability`.
2. Выбор: **самый ранний** `plannedStart`; при равенстве — раньше `plannedEnd`.
3. Якорь = max(`availableAt` станка, предшественник по технологии, `orderStart`), затем `nextWorkStart` (смена).
4. Операция целиком должна поместиться в одно окно смены (`nextShiftStartFittingDuration`).

### 7.1. Предшественник (`previousTaskEnd`)

Для **завершённых** операций в расчёте используется **`actualEnd`**, для плановых — `plannedEnd`.

- **sequence = 0, unit 0:** `orderStart`.
- **sequence = 0, unit k > 0:** конец той же операции у unit `k − 1`.
- **sequence > 0, unit 0:** см. режим overlap в §5 (`PIPELINE` / `PER_UNIT` / `NONE`).
- **sequence > 0, unit k > 0:** `max(end(op N, k), end(op N+1, k−1))` на той же op.N+1.

Ограничение «вся партия op.N на одном станке» для op.N+1 на **другом** станке при overlap **не** применяется; для op.N+1 на **том же** станке — применяется (`latestEarlierSharedMachineBatchEnd` только по назначениям на `targetMachineId`).

## 8. Смены и календарь

- Часовой пояс: **Europe/Moscow** (`FactoryZone.ZONE`).
- Работа только в `workWindows` группы станка; длительности `PT…` — **чистое рабочее время** (`ShiftCalendar.addWorkDuration`).
- Не начинать операцию, если она не уместится до конца текущего окна смены.
- Группы смен и `setupMinutes` можно менять после старта через **`PUT /machine-groups/{groupId}`** (влияет на **новые** назначения; уже записанный план не пересчитывается автоматически).

## 9. Закрытие смены и перепланирование

### 9.1. Контекст и «просрочка» (`GET /shifts/context`)

- **`pendingShifts`** — прошедшие смены группы, у которых `shiftEnd` > `lastClosedShiftEndByGroup[groupId]` (отсчёт от `factoryStartedAt`, если закрытий ещё не было).
- **`stale: true`** — есть хотя бы одна такая смена.
- **`activeShift`** — первая pending (самая старая незакрытая) **или**, если pending нет, текущая идущая смена группы с работой в окне.
- В таблице формы: агрегаты **`plannedCount`** по `(machineId, taskId)` для `PLANNED` рабочих операций, пересекающих окно `[shiftStart, shiftEnd]`; по умолчанию «сделано» = всё запланированное.
- **`closeRows`** — плоский список всех строк для закрытия **всех** pending-смен (с полем `groupId`); используется кнопкой «Закрыть все смены».

### 9.2. Автозакрытие пустых смен (`ShiftAutoCloseService`)

При `GET /shifts/context` и `PUT /time`: для каждой pending-смены, в окне которой **нет** `PLANNED` работы группы, в `lastClosedShiftEndByGroup` записывается конец смены **без переплана**. Цикл повторяется, пока есть такие смены.

### 9.3. Закрытие с фактами (`POST /shifts/close`)

**Рекомендуемое тело в HTML** — закрытие **всех** незакрытых смен сразу (`closeAllPendingShifts: true`). В таблице — все станки всех групп из `GET /shifts/context` → `closeRows`; для каждой строки обязателен `groupId`, `machineId`, `taskId`, `completedCount`. Переплан **один раз** после применения всех фактов.

```json
{
  "closeAllPendingShifts": true,
  "shiftEnd": "2026-05-22T17:00:00Z",
  "machineTaskCounts": [
    { "groupId": "cnc", "machineId": "ТОКАР-ЧПУ-02", "taskId": "черновая-токарка", "completedCount": 7 },
    { "groupId": "cnc", "machineId": "ФРЕЗ-ЧПУ-01", "taskId": "черновая-фрезеровка", "completedCount": 4 }
  ],
  "idleBlocks": []
}
```

**Одна группа** (API, без `closeAllPendingShifts`):

```json
{
  "groupId": "cnc",
  "shiftStart": "2026-05-22T05:00:00Z",
  "shiftEnd": "2026-05-22T17:00:00Z",
  "machineTaskCounts": [
    { "machineId": "ТОКАР-ЧПУ-02", "taskId": "черновая-токарка", "completedCount": 7 }
  ],
  "idleBlocks": []
}
```

**Развёртывание** (`ShiftFactResolver`): в окне смены все `PLANNED` рабочие операции (не `переналадка`) группируются по `(machineId, taskId)`, сортировка: `plannedStart` → `priority` заказа (↓) → `orderId` → `unitIndex`. Первые **N** из `completedCount` → `COMPLETED` (`actualEnd` = min(`plannedEnd`, `shiftEnd`)), остальные в этой группе → `CANCELLED`.

**Поштучный режим** — поле `operations[]` (как раньше): `orderId`, `partId`, `unitIndex`, `taskId`, `completed`, опционально `actualStart` / `actualEnd`.

**Простои** — `idleBlocks[]` → `machineBlocks` в JSON; сдвигает план на этом станке при переплане.

### 9.4. Переплан (`ReplanService`)

После записи **всех** фактов (при `closeAllPendingShifts` — по каждой незакрытой смене каждой группы, затем **один** вызов переплана):

1. Собирается множество **затронутых** `orderId` (из фактов и заказов на станках с простоями).
2. Для каждого такого заказа (в порядке `priority` в store): все **`PLANNED`** → `CANCELLED`.
3. **`COMPLETED`** не меняются; переналадки к завершённым операциям тоже закрываются по факту.
4. Хвост достраивается `GreedyScheduler` с **`ReplanContext`**: по возможности тот же станок, что был у снятого `PLANNED`.
5. Для каждой закрытой группы: `lastClosedShiftEndByGroup[groupId]` = конец её смены.
6. Ответ содержит `replannedOrderIds`, `orderReadyAt` (пересчёт `readyAt` по затронутым заказам).

**Важно для отчётов:** частичное закрытие **ранних** операций (например, черновая токарка вала) **часто не меняет** `readyAt` всего заказа, если критический путь — другая деталь (например, корпус с расточкой). `readyAt` — прогноз **полного** окончания заказа, а не «процент смены».

## 10. Переналадка

- Длительность — **`setupMinutes` / `setupDuration`** группы станков (в JSON обычно `setupMinutes`; при загрузке нормализуется в `Duration`).
- Нужна, если на станке последняя **рабочая** операция была с **другим** `taskId` (тип операции: черновая → чистовая, фрезеровка → чистовая и т.д.) или станок ещё пуст.
- Тот же `taskId` подряд (следующая штука той же операции) → переналадка **0**.
- Смена только `partId` при том же `taskId` на станке в MVP не встречается (разные детали — разные маршруты); при смене детали **или** типа операции — переналадка.
- При расчёте «последней операции на станке» учитываются только **активные** назначения (`PLANNED` / `COMPLETED`); **отменённые** после частичного закрытия смены не влияют на переналадку.
- В HTML/JSON расписания **отменённые** назначения не показываются; в прошлой смене остаются только фактически выполненные операции.
- Запись с `taskId = "переналадка"` перед работой на той же `unitIndex`; **не** увеличивает `sequence` маршрута.

---

## 11. Занятость станка

- `availableAt` = max(прежнее, `plannedEnd` последней операции на станке).
- `MachineStateSync`: если `plannedEnd` в прошлом относительно «сейчас», станок свободен с **текущего момента**.
- `IDLE` / `BUSY` — автоматически; `DOWN` / `MAINTENANCE` / `SETUP` — только вручную (`PATCH /machines/{id}`).

---

## 12. Метрики готовности

Учитываются только назначения со статусом **не** `CANCELLED` (`AssignmentFilters.active`).

| Показатель | Определение |
|------------|-------------|
| `effectiveStart` / `effectiveEnd` | `actual*` если есть, иначе `planned*` |
| `partReadyAt` | max `effectiveEnd` по всем назначениям детали в заказе (включая переналадки) |
| `readyAt` заказа | max `effectiveEnd` по **всем** активным назначениям заказа |
| «Резерв» в HTML (`slackMinutes`) | `readyAt` − `partReadyAt` (минуты) |

## 14. Ограничения MVP

1. Жадный алгоритм — не глобальный оптимум.
2. Новый заказ без закрытия смены **не двигает** старые `PLANNED`.
3. Нет склейки партий между заказами.
4. Нет учёта операторов, инструмента, брака (кроме `machineBlocks` и фактов смены).
5. План не исполняется на станках автоматически — модель прогноза.

---

## 15. Схема потока

```
POST /orders (createdAt = сейчас, orderId опционально)
    ↓
Пока не все детали запланированы → readyWork → станок → смены
    ↓
Проверки: partReadyAt в заказе, readyAt между заказами
    ↓
Сохранение в schedule.json

--- позже ---

GET /shifts/context (автозакрытие пустых смен)
    ↓
POST /shifts/close (counts или operations + idleBlocks)
    ↓
Факты COMPLETED/CANCELLED → replan затронутых заказов → orderReadyAt
```
