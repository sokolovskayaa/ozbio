# Правила планировщика (MVP)

Документ описывает **фактическую логику кода** (май 2026): жадный планировщик `GreedyScheduler`, метрики `OrderProgress` / `TaskReadiness`, сохранение в `data/schedule.json`.

**Не в этом MVP:** смены и `workWindows`, закрытие смены, переплан, overlap пакетов, симуляция времени (`PUT /time`), изменение статуса станка по API, дублирующие станки.

---

## 1. Обзор

| Компонент | Роль |
|-----------|------|
| `POST /orders` | Добавить заказ и спланировать его |
| `GET /schedule` | JSON или HTML отчёт |
| `GreedyScheduler` | Жадное назначение операций на станки |
| `JsonScheduleRepository` | Загрузка/сохранение snapshot (тесты, legacy JSON) |
| `SchedulePageRenderer` | HTML через Thymeleaf + статика |

Планировщик работает **круглосуточно** (24/7). Длительности в справочнике (`PT70M` и т.д.) — календарное время на станке.

---

## 2. Данные

Состояние планировщика в **PostgreSQL** (Liquibase). Эталонный JSON: `data/schedule.json.example` (для тестов).

| Таблица | Смысл |
|---------|--------|
| `factory_state` | `factory_started_at` |
| `machine_group` | Группы, `setup_minutes` |
| `machine`, `machine_capability` | Парк станков |
| `part_definition`, `part_task` | Справочник деталей |
| `schedule_order`, `order_part`, `order_part_task` | Заказы |
| `assignment` | Назначения |

---

## 3. Заказ и очередь

- `orderId` опционально; автономер `З-ГГГГ-NNNN` (год по часовому поясу завода, Москва).
- `createdAt` = **текущее системное время** (`SystemCurrentTimeProvider`).
- Очередь заказов: **FIFO** по `createdAt` (`OrderPriorities.QUEUE_ORDER`).
- Новый заказ **не должен увеличивать** `readyAt` более ранних заказов в очереди.

---

## 4. Пакетное планирование по операциям

Для каждой детали в заказе:

1. Все штуки проходят **операцию 0**, затем все — **операцию 1**, …
2. Внутри одной операции штуки идут по `unitIndex` 0 → 1 → 2 …
3. Операция **N+1** для любой штуки не начинается, пока **все** штуки не запланированы или выполнены по операции **N** (классический пакет, без overlap).

**Приоритет детали** (`partDefinitions.priority`) действует **только внутри одного заказа**: при параллельных станках более приоритетная деталь не должна получить более поздний `partReadyAt` из-за менее приоритетной.

---

## 5. Выбор станка и переналадка

- Станок подбирается по `requiredCapability` задачи.
- Если несколько станков с одной capability — выбирается с минимальным `availableAt`.
- **Переналадка** (`SetupPlanner`): перед первой операцией на станке и при **смене `taskId`** на этом станке; длительность из `setupMinutes` группы.
- В HTML переналадка — оранжевый отрезок `переналадка`.

Демо-парк (`DemoFactoryCatalog`): один станок на capability (6 станков, 3 группы).

---

## 6. Готовность и метрики

| Метод | Класс | Смысл |
|-------|--------|--------|
| `readyAt` | `OrderProgress` | Максимальный `effectiveEnd` активных назначений заказа |
| `partReadyAt` | `OrderProgress` | То же для детали |
| `orderStart` | `OrderProgress` | max(`createdAt`, `factoryStartedAt`, `now`) |
| `readyWork` | `TaskReadiness` | Следующая незапланированная операция |
| `isTaskReady` | `TaskReadiness` | Можно ли начать операцию (пакет + предшественники) |
| `previousTaskEnd` | `TaskReadiness` | Ограничение по технологической цепочке и станку |

`readyAt` заказа — прогноз **полного** завершения (критический путь), не «процент выполнения».

## 7. API

| Метод | URL | Описание |
|--------|-----|----------|
| POST | `/orders` | Добавить заказ |
| GET | `/schedule` | JSON (по умолчанию) |

## 8. HTML-отчёт

- Шаблон: `templates/schedule.html`
- Стили/скрипты: `static/css/schedule.css`, `static/js/schedule-actions.js`, `static/js/schedule-filters.js`
- Секции: управление (форма заказа), заказы, парк станков, группы, **расписание по станкам** (Gantt), справочник
- Фильтры и пресеты масштаба: **День / 3 дня / Неделя / …** (не «сменные окна»)

---

## 9. Ограничения MVP

1. Нет учёта смен — план 24/7.
2. Нет фактов производства и переплана.
3. Нет overlap: чистовая на том же станке — только после всей черновой партии.
4. Нет API простоя/DOWN станка (статус в JSON только для отображения).
5. Склейка одинаковых деталей из разных заказов без переналадки — **не** выполняется.
6. «Сейчас» на странице — системное время; нельзя перемотать день через API.
