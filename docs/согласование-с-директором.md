# Ручное тестирование и согласование с директором

Пошаговый сценарий встречи для проверки MVP планировщика. Эталон поведения: [правила-планировщика.md](правила-планировщика.md).

| Параметр | Значение |
|----------|----------|
| Длительность | 60–90 мин (ядро ~45 мин) |
| Экран | **http://localhost:8080/schedule?format=html** |
| Не использовать | Файл `schedule-demo.html` с диска — форма закрытия смены без сервера не работает |

**Быстрая подготовка:** `./scripts/prep-director-demo.sh`

---

## Подготовка оператора (до встречи)

- [ ] Сервер остановлен → `./scripts/reset-demo-data.sh` → снова `mvn -q exec:java` (или `scheduler.Main`)
- [ ] `./scripts/prep-director-demo.sh` — без ошибок (время 08:00, заказов 0)
- [ ] В браузере открыт http://localhost:8080/schedule?format=html
- [ ] На втором мониторе: [правила-планировщика.md](правила-планировщика.md) (§4, §9, §12)

**Опционально (формат «Презентация»):** `./scripts/demo.sh` до шага 18:00, на встрече начать с [сценария 3](#сценарий-3-расписание-по-станкам).

### Вступление для директора (~30 сек)

- Это **прогнозный план**, не MES: мастер **закрывает смену** и указывает, сколько сделано.
- **Готовность заказа** — когда закончится **весь** заказ (самая долгая деталь), не «процент смены».
- Время в демо **симулируется** (можно перемотать день).
- **Параллельные пакеты (overlap):** только при **`scheduling.overlapBatches: true`** (в шаблоне дублей станков уже вкл.; базовый `schedule.json.example` — выкл.). Между разными станками (фреза → расточка); при двух токарных — чистовая на втором может начаться до конца всей черновой на первом. На **одном** токарном вал: сначала все 8 черновых, потом чистовая.

## Сценарий 1. Первый заказ: две детали, пакет по операциям

**Цель:** согласовать «сначала все штуки — операция 1, потом все — операция 2» и параллельность вал/корпус.

### Действия

1. «Добавить заказ»: **вал-буровой 8 шт.** + **корпус-бура 8 шт.** (без номера — автономер `З-2026-…`, или заранее `З-2026-0142`).
2. Таблица заказа: **готовность**, **резерв** (у корпуса резерв меньше).
3. Операции: у вала сначала **8× черновая токарка**, потом чистовая, шлифование; у корпуса — фрезеровка → расточка → чистовая.

### Ожидание

| № | Ожидание | § |
|---|----------|---|
| 1.1 | План на **все 8+8** штук | §4, §6 |
| 1.2 | Все **8** черновых токарок **до** первой чистовой | §4.1 |
| 1.3 | Вал и корпус **параллельно** на разных станках | §5 |
| 1.4 | Готовность заказа ≈ конец цепочки **корпуса** | §12 |
| 1.5 | **Overlap** (нужен `overlapBatches: true`): расточка корпуса на РАСТОЧ-03 **начинается до** окончания всех 8 черновых фрезеровок на ФРЕЗ-ЧПУ-01 | §5 |

**Вопрос директору:** «Сначала весь черновой проход партии на одном станке, но следующая операция на **другом** станке может стартовать раньше — так и задумано?»

## Сценарий 3. Расписание по станкам

**Цель:** смены, переналадка, загрузка.

### Действия

1. Блок **«Расписание по станкам»**.
2. Пресеты **Смена / День / 3 дня**.
3. **ТОКАР-ЧПУ-02** — черновая токарка, затем **оранжевая переналадка**, затем чистовая (смена типа операции).
4. **ФРЕЗ-ЧПУ-01** — корпус, черновая фрезеровка; **РАСТОЧ-03** — расточка **параллельно** (overlap), пока фреза ещё занята.
5. Фильтр: заказ 0142 → «корпус-бура» — расточка **начинается раньше**, чем конец последней черновой фрезеровки.
6. Фильтр: «вал-буровой».
7. Статусы **Занят / Свободен** (не путать с DOWN).

### Ожидание

| № | Ожидание | § |
|---|----------|---|
| 3.1 | Работа в **сменных** окнах, ночь пустая | §8 |
| 3.2 | Длинная операция уходит на **след. день**, не обрезается | §8 |
| 3.3 | Переналадка **перед** первой операцией и **между типами** (черновая → чистовая токарка) | §10 |
| 3.4 | Подписи по-русски, заказ/деталь читаемы | UI |
| 3.5 | **Overlap:** расточка корпуса пересекается по времени с черновой фрезеровкой | §5 |

**Вопрос директору:** «Так удобно смотреть загрузку на неделю?»

## Сценарий 4. Конец дня и закрытие смены

**Цель:** факт смены и переплан.

### Действия

1. Время → **18:00**.
2. **«Закрытие смены»**: баннер просрочки (если есть).
3. Таблица **в плане / сделано** (по умолчанию = всё).
4. Пример факта: **черновая токарка 7**, **черновая фрезеровка 4** → «Закрыть смену».
5. После reload: **выполнено** на шкале, хвост **перепланирован**.
6. **readyAt** может **не измениться** — критический путь на корпусе (расточка и далее).

### Ожидание

| № | Ожидание | § |
|---|----------|---|
| 4.1 | Только работа **в окне смены** ЧПУ | §9.1 |
| 4.2 | Можно **меньше**, чем в плане | §9.3 |
| 4.3 | Сообщение о **перепланировании** | §9.4 |
| 4.4 | **readyAt** может не сдвинуться | §9.4, §12 |
| 4.5 | Пустые смены закрываются **сами** | §9.2 |

**Вопрос директору:** «Достаточно вводить количество по типу операции на станке?»

  -d '{"orderId":"З-2026-0150","parts":[{"partId":"вал-буровой","quantity":1}]}'
